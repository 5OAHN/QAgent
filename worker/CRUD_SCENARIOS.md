# QAgent CRUD 테스트 시나리오 (견고한 설계)

## 개요
TodoMVC React를 대상으로 한 완전한 CRUD 사이클 검증.
각 시나리오는 **단계 분리, 명시적 성공 조건, 스냅샷 갱신 강제**를 따름.

---

## 📋 시나리오 1: CREATE (항목 생성)

**목표**: 입력창에서 새 항목을 추가하고 목록에 표시되는지 확인

```
1. 입력창에 '구매하기'를 입력한다
2. Enter 키를 누른다
3. 목록에 '구매하기'가 표시되고 '1 item left'가 보이면 성공
```

**성공 조건**:
- 입력창이 비워짐 (새 항목 추가 신호)
- '구매하기' 텍스트가 화면에 표시됨
- '1 item left' 카운터 표시

**실패 시 검사**:
- ref 유효성: 입력창 ref가 현재 스냅샷에 존재
- 초기 상태: 이미 항목이 있다면 pass (기존 데이터는 무시)

---

## 📋 시나리오 2: READ (목록 확인)

**목표**: 방금 추가한 항목이 정말 목록에 있는지 2중 확인

```
1. 현재 화면에 '구매하기' 항목이 목록에 표시되는지 확인한다
2. 항목 옆의 '1 item left'가 보이면 성공
```

**성공 조건**:
- '구매하기' 텍스트가 pageText에 포함됨
- '1 item left' 또는 '1개 항목' 같은 카운터 텍스트 표시

**용도**: Step 1의 결과가 유지되는지 검증 (DOM 재렌더링 후에도 살아있는지)

---

## 📋 시나리오 3: UPDATE (항목 편집 - 더블클릭)

**목표**: 더블클릭으로 편집 모드에 진입

```
1. '구매하기' 항목의 텍스트를 더블클릭한다
2. 입력 상태의 창이 나타나면 성공
```

**성공 조건**:
- 편집 input 요소가 현재 스냅샷에 나타남
- input의 value가 '구매하기'를 포함

**주의점**:
- dblclick 후 **반드시 새 스냅샷을 만들어서** 편집 input의 새 ref를 획득
- 이전 스냅샷의 ref로 fill하면 stale ref 에러

---

## 📋 시나리오 4: UPDATE (항목 편집 - 텍스트 변경)

**목표**: 편집 모드에서 텍스트를 변경하고 Enter로 저장

```
1. 편집 입력창의 텍스트를 '구매하기-긴급'으로 변경한다
2. Enter 키를 누른다
3. 목록으로 돌아가고 '구매하기-긴급'이 표시되면 성공
```

**성공 조건**:
- '구매하기-긴급' 텍스트가 목록에 표시됨
- 편집 모드가 닫혀 있음 (input 태그가 화면에서 사라짐)

**에러 방어**:
- fill 실패 시: 키보드로 폴백 (Control+A → type)
- ref 무효화: 새 스냅샷의 ref만 사용

---

## 📋 시나리오 5: DELETE CHECK (항목 완료 표시)

**목표**: 체크박스를 클릭해서 항목을 "완료"로 표시

```
1. '구매하기-긴급' 항목의 왼쪽 체크박스를 클릭한다
2. 항목에 취소선이 그어지고 '0 items left'가 보이면 성공
```

**성공 조건**:
- 체크박스가 ✓ 상태 (checked=true)
- 항목 텍스트에 취소선 CSS 적용됨 (또는 화면에 시각적 변화)
- 카운터가 '0 items left' 또는 '0개'로 변경됨

**포인트**:
- 항목이 목록에서 사라지지 않음 (완료로 표시만, 삭제 아님)

---

## 📋 시나리오 6: DELETE (항목 삭제)

**목표**: X 버튼을 클릭해서 항목을 완전히 제거

```
1. 완료된 '구매하기-긴급' 항목의 우측 X 버튼(삭제)을 클릭한다
2. 목록에서 항목이 완전히 사라지면 성공
```

**성공 조건**:
- 항목이 DOM에서 제거됨
- '구매하기-긴급' 텍스트가 pageText에 포함되지 않음
- 목록이 비어 있음

**확인**:
- 초기 상태 복원: 항목 0개

---

## 🔄 전체 사이클 (한 번에 테스트)

### 옵션 A: 6개 시나리오를 개별로 실행 (권장 — 병렬 실행 가능)

```bash
# 각 시나리오를 독립적인 테스트로 제출
# 각각 새 브라우저 세션에서 실행되므로 서로 영향 없음
```

**장점**: 개별 실패 원인 파악 용이, 병렬 실행 가능
**단점**: 6번 서버 호출

### 옵션 B: 하나의 긴 시나리오 (순차 실행)

```
1. 입력창에 '구매하기'를 입력하고 Enter를 누른다. 목록에 표시되면 성공
2. 현재 화면에 '구매하기'가 보이고 '1 item left'가 표시되면 성공
3. '구매하기' 항목을 더블클릭해 편집 모드로 진입한다. 입력창이 나타나면 성공
4. 편집 입력창의 텍스트를 '구매하기-긴급'으로 변경한다. Enter를 누르면 성공
5. 목록으로 돌아가고 '구매하기-긴급'이 표시되면 성공
6. '구매하기-긴급' 항목의 체크박스를 클릭한다. 취소선이 그어지고 '0 items left'가 보이면 성공
7. 우측 X 버튼을 클릭해 항목을 삭제한다. 목록이 비어있으면 성공
```

**장점**: 한 번의 호출, 실제 사용자 흐름과 동일
**단점**: 한 단계 실패 시 뒤의 모든 단계도 실패

## ⚙️ 기술적 가이드 (AI 에이전트용)

### 각 단계에서 꼭 해야 할 것

1. **Step 입구마다 새 스냅샷**: 화면이 갱신되면 ref가 바뀌므로 항상 현재 ref 재획득
2. **fill은 한 번에**: 기존 값을 자동으로 지우므로 Control+A/Delete는 불필요
3. **reasoning은 짧게**: 1-2문장으로, 장문은 응답이 잘려서 turn이 무효 처리됨
4. **타임아웃 시 방향 전환**: 같은 ref 3회 이상 타임아웃되면 click 후 press로 폴백

### 발생 가능한 에러와 대처

| 에러 | 원인 | 대처 |
|------|------|------|
| `action 필드 누락` | 토큰 잘림 (보통 reasoning이 길었음) | reasoning을 1문장으로 단축 |
| `fill timeout` | React 재렌더링으로 input이 여전히 detach 상태 | click → Cmd+A → type으로 폴백 |
| `ref=N 요소가 DOM에 없음` | 이전 턴의 ref가 재렌더링으로 무효화됨 | 새 스냅샷을 만들고 새 ref 사용 |
| 연속 3회 실패 | 단순 반복이 통하지 않는 상황 | 접근을 바꿈: 다른 요소 사용, 다른 키 입력, 이전 단계 재진입 |

## 📊 기대 성과

**성공 시**:
- ✅ 6/6 단계 또는 7/7 단계 통과
- 전체 소요 토큰: ~3,000~4,000 (3~4건의 API 호출)
- 소요 시간: ~30~60초

**실패 시**:
- 실패 단계 명시 + AI 에이전트의 reasoning 로그 포함
- 스냅샷 + 에러 메시지로 원인 파악 용이
- 해당 단계 이전부터 재시도 가능

## 🎯 왜 이렇게 설계했나

### 1. 단계 분리의 중요성
- 각 단계가 1~2개의 사용자 액션으로 제한
- "한 단계 = 한 번의 스냅샷 + 한 번의 에이전트 호출"
- 이렇게 하면 어느 단계가 깨졌는지 명확함

### 2. 명시적 성공 조건
- "성공하면 ~가 보여야 한다"를 구체적으로 명시
- 에이전트가 "이게 성공인가?"를 객관적으로 판단 가능

### 3. 스냅샷 갱신 강제
- 특히 dblclick, fill 같이 DOM을 바꾸는 액션 후에는 "새 스냅샷을 만들어라"고 명시
- 이전 턴의 stale ref로 낭비하는 8초를 막음

### 4. 에러 방어 내장
- 시나리오 자체가 "fill 실패 → type 폴백"을 예상하고 작성
- "연속 실패 시 다른 방법"을 미리 제시

## 💡 확장 시나리오 (고급)

### 다중 항목 CRUD
```
1. 3개 항목을 순차적으로 추가한다: '장보기', '청소하기', '코드 리뷰'
2. 중간 항목을 완료 표시한다
3. 첫 번째 항목의 이름을 변경한다
4. 마지막 항목을 삭제한다
5. 최종 상태: 2개 항목, 1개 완료
```

### 엣지 케이스
```
1. 입력창에 공백만 입력하고 Enter를 누른다 → 입력되지 않아야 함
2. 같은 이름의 항목 2개를 추가한다 → 둘 다 추가되어야 함
3. 모든 항목을 완료 표시한다 → '0 items left'로 변경
4. 완료한 항목을 다시 체크 해제한다 → 상태 복구
```
